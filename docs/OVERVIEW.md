# Signal System — High-Level Overview
Last updated: 2026-06-26

---

## Что это и зачем

Сервис мониторинга сигналов для B2B консультантов (executive search, рекрутинг, продажи).
Система находит компании в нужный момент — когда у них что-то происходит — и готовит клиенту
контакт + контекст + текст письма. Клиент открывает трекер и видит готовые лиды.

Первый клиент: Philippe Bosquillon (executive search, food industry, DE/FR/NL/BE/LU).
Цель: 5-10 клиентов на одной инфраструктуре, разные конфиги.

Это часть более широкой outreach системы: раз в месяц-полтора делается blitz по всему рынку,
плюс постоянный мониторинг сигналов между бластами.

---

## Два типа сигналов

**HIRING** — компания разместила вакансию на управленческую должность.
Источник: Apify акторы на job boards (LinkedIn, StepStone, Xing, Cadremploi, Indeed).
Логика: если нанимают Werksleiter или Directeur Usine — расширяются или меняют руководство.
Окно возможности: 60 дней с публикации.

**NEWS** — что-то произошло с компанией: M&A, смена CEO, расширение, инвестиция.
Источник: Exa Monitors (33 штуки на клиента, каждые 7 дней).
Логика: outreach с контекстом в 3x эффективнее холодного. "Видел новость X" вместо "хочу познакомиться".
Окно возможности: зависит от типа (C-Level — 30 дней, M&A/Expansion — 60-90 дней).

Оба типа проходят один и тот же pipeline. Разница только в ICP правилах и LLM промптах.

---

## Как сигнал проходит через систему

```
Источник (Apify / Exa webhook)
  → raw_signals        все сырые данные, без фильтрации
  → ICP фильтр         отрасль + размер компании + уровень должности + staleness
  → companies          дедупликация по linkedin_url / domain
  → contacts           Blitz API: поиск ЛПР + email
  → LLM scoring        score + narrative (контекст) + angle (угол захода)
  → signals            финальные данные в Supabase
  → трекер             клиент видит готовый лид
```

Всё что не прошло ICP — остаётся в raw_signals со статусом filtered_out и причиной.
Не теряем данные, можем посмотреть почему что-то отфильтровалось.

## Три сценария использования

**Auto** — сигнал прошёл пайплайн, контакт найден, письмо сгенерировано,
автоматически пушится в PlusVibe секвенсер. Клиент не участвует.
Подходит для WARM/COLD сигналов (score 2-6).

**Copy-paste** — клиент заходит в трекер, видит карточку компании с готовым текстом,
копирует и отправляет сам. Минимум решений, максимум контекста.
Подходит для HOT/PRIORITY сигналов (score 7-10) где важен личный подход.

**Hybrid** — топ-тир сигналы клиент обрабатывает сам, остальные идут в секвенсер автоматически.
Настраивается через порог score в client config.

## Мульти-клиент: как это устроено

База данных: все таблицы имеют client_id. Один Supabase, одна схема (signal_monitoring),
данные разных клиентов физически в одних таблицах но логически изолированы.

Конфиги: один JSON файл на клиента в pipeline/clients/{slug}.json.
Содержит: ICP фильтры, источники сигналов, правила скоринга, настройки copy, sequencer config.
Скрипты читают конфиг через --client флаг. Новый клиент = новый JSON файл.

Exa мониторы: каждый клиент имеет свои мониторы на api.exa.ai.
Маппинг monitor_id → тип сигнала хранится в exa/clients/{slug}.json.
Webhook определяет клиента по токену в URL.

Apify: один аккаунт (KAD2), разные конфиги акторов per клиент.

Принцип конфигов: скрипты всегда читают с дефолтом — если поля нет в конфиге клиента,
берётся дефолтное значение. Новые поля можно добавлять в шаблон не ломая старых клиентов.

## Ключевые архитектурные решения

**Next.js вместо отдельного бэка.**
API routes в Next.js достаточно. Нет смысла в Express для 3-4 эндпоинтов.
Если вырастет — можно вынести, но не сейчас.

**Supabase self-hosted на VPS.**
Полный контроль, нет лимитов на строки/запросы, стоит $0 дополнительно.
Один инстанс на все проекты. Схемы изолируют проекты (signal_monitoring, outreach и т.д.).

**raw_signals хранит всё до фильтрации.**
Дорого потерять контекст почему что-то отфильтровалось. Хранить дёшево.
ICP фильтр — не истина в последней инстанции, правила меняются.

**Typed колонки только для того что фильтруем/сортируем.**
score, pub_date, source, status, signal_type — колонки.
Всё нестабильное и экспериментальное — в meta jsonb. Когда поле устаканится — переносим в колонку.

**Client config в JSON, не в DB.**
Меняется редко, только Leo настраивает, должен быть в git (история изменений).
В DB переедет когда клиенты сами захотят настраивать через UI — не сейчас.

**Exa маппинг в JSON файле, не в коде.**
monitor_id → signal_type это данные, не логика. Хранить в коде = деплой при добавлении клиента.
Хранить в файле = положил файл, перезапустил.

**Аутентификация: ?client=slug в URL.**
Для 1-2 клиентов достаточно. Полноценный auth добавляется когда появится реальная потребность.

## Что намеренно не строим сейчас

- Auth система и личные кабинеты — не нужно для 1-2 клиентов
- Онбординг UI — Leo настраивает руками, это быстрее чем строить UI
- /monitoring страница — в настройках если понадобится, не в основном UI
- Телеграм уведомления — всё видно в трекере
- Автоматический рескоринг устаревших сигналов — будущая фича
- Карьерные страницы через changedetection.io — на паузе, сложность vs выхлоп неясна

---

## Источники которые есть и могут добавляться

Сейчас: Exa (новости) + Apify (job boards).
Потенциально: Crunchbase (инвестиции), LinkedIn Sales Navigator API, RSS отраслевых изданий,
career pages мониторинг, любой другой скрапер через Apify.

Принцип: новый источник = новый нормализатор в pipeline/lib/normalize/ + запись в raw_signals.
Дальнейший pipeline (фильтр → enrich → score → push) одинаковый для всех.

## Стек

| Слой | Технология | Почему |
|------|-----------|--------|
| Фронтенд | Next.js 14 + TypeScript + Tailwind + shadcn/ui | App Router + API routes в одном месте |
| БД | Supabase self-hosted (Postgres 15) | Self-hosted = без лимитов и затрат |
| Скрапинг | Apify | Готовые акторы, не нужно поддерживать скраперы |
| Новости | Exa Monitors | 33 запроса, автоматически каждые 7 дней, $2.4/месяц |
| Обогащение | BlitzAPI | Waterfall поиск контактов, лучший из тестированных |
| Email validation | BounceBan | Manual upload flow, дёшево |
| LLM | OpenRouter → gpt-oss-120b | Гибкость смены модели, кэш дорогих запросов |
| Sequencer | PlusVibe | Уже используется в outreach системе |
| Хостинг | Netcup VPS + Coolify | Один VPS на все проекты, auto-deploy |
| Оркестрация | Node.js + crontab | Plain cron, без BullMQ/Celery для 10 задач в неделю |
